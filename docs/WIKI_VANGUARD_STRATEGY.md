# The Vanguard Strategy: Adversarial Tri-Agent Gauntlet ⚔️

In 2026, verification is the primary bottleneck in software engineering. Project Claudia resolves this through an adversarial model where code is treated as a liability until it survives our specialized gauntlet.

## 🏗️ The Agents

### 1. The Builder (Generator)
The Builder's role is to translate requirements into a functional code delta. It is grounded via **Model Context Protocol (MCP)**, allowing it access to Jira issues, GitHub state, and local documentation.

### 2. The Adversary (Hacker)
The Adversary is programmed with a library of real-world vulnerabilities (SQLi, XSS, Proto-Pollution) and specialized logic-bypass strategies. Its mission is to **break the Builder's code**. 

### 3. The Arbiter (Judge)
The Arbiter evaluates the duel. It prevents "lazy security" (e.g., the Builder just adding comments instead of fixes) and ensures that the final PR diff is both functional and secure.

## 🛡️ The Life of a Verification Duel
1. **Challenge**: Requirements are fed into the system.
2. **Draft**: The Builder generates a candidate implementation.
3. **Attack**: The Hacker generates multiple adversarial test cases.
4. **Execution**: The code is run in the **Universal Sandbox**.
5. **Certification**: If the code survives the attack and meets requirements, the Arbiter certifies the Unified PR Diff.
